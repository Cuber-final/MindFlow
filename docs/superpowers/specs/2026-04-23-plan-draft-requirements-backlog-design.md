# Plan-Draft Requirements Backlog Design

> 日期：2026-04-23
> 状态：approved-in-chat
> 目标：把 [2026-04-22-plan-draft.md](../plans/2026-04-22-plan-draft.md) 中的后续开发规划，拆成可持续维护的需求条目，并归档进统一需求池 [requirements-backlog.md](../../requirements-backlog.md)

## 设计结论

- [requirements-backlog.md](../../requirements-backlog.md) 继续作为统一需求入口
- `plan-draft` 不直接替代需求池，而是作为需求拆解来源
- 可直接开发的条目写成普通需求，状态标记为 `planned`
- 需要先做产品边界判断的条目，使用 `DEC` 分类，状态标记为 `parked`

## 为什么这样拆

`plan-draft` 给的是“下一阶段优先级排序”，不是稳定 backlog。

如果直接把它当 backlog 用，会有两个问题：

1. 排序项过大，后续难以继续拆 plan
2. 决策问题和开发问题混在一起，容易让需求池失真

因此需要多一步：

- 先按主题拆成需求条目
- 再按类型归档进需求池

## 需求拆解规则

### 1. 调度与任务可靠性

归入 `OPS`：

- 调度链路修通
- 任务状态与失败原因可观测
- 任务 / 日志页
- 刷新频率设置接入 UI

### 2. 检索与内容输出能力

归入 `FLOW`：

- 文章搜索
- 多维筛选
- 导出能力
- 搜索 / 筛选结果的阅读闭环

### 3. 平台化数据模型与治理

以 `DATA` 为主，必要时补 `SRC`：

- Source 治理字段扩展
- Entry / Job 状态字段扩展
- 前后端契约同步
- Sources 页治理字段操作入口

### 4. 决策型需求

归入 `DEC`：

- 是否引入统一时间线 / 搜索结果阅读面

这类条目先保留在需求池，但不直接进入开发计划。

### 5. 文章级 AI 加工增强

归入 `AI`：

- 关键词提取
- 语言识别
- 可选翻译
- AI processing 结果在阅读面的消费方式

## 对需求池结构的影响

[requirements-backlog.md](../../requirements-backlog.md) 需要补充两项规则：

1. 增加 `DEC` 分类
2. 明确“决策型需求默认使用 `parked` 状态”

此外，需求池中的条目按两层组织：

- 已录入的交互 / Source 体验需求
- 基于 `2026-04-22-plan-draft.md` 拆解的后续开发需求

## 范围控制

这次只做需求拆解与归档，不做：

- 具体实现方案
- 接口设计细节
- 数据库字段级设计
- 页面交互稿
- 实施计划拆解

后续如果某个主题进入开发，再单独写 plan。
