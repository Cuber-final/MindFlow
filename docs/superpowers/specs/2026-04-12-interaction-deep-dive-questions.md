# 深挖需求问题清单

| 问题ID | 关联交互 / Gap ID | 优先级 | 当前冲突 / 不确定点 | 需要产品决策的问题 | 决策选项 | 推荐 |
|---|---|---|---|---|---|---|
| `DDQ-01` | `Newsletter / 日期选择器 / 今天`；`IA-BE-001` | P1 | `GET /api/digests` 当前返回裸数组，但前端按 `items` DTO 读取。 | 历史简报列表应被定义成“轻量数组列表”还是“分页 DTO 列表”？ | A. 保持后端裸数组，前端按数组消费并在 TECH 中明示。<br>B. 后端改为 `{ items, total, limit, offset }`，前端统一走 DTO。 | **A（推荐）**：最贴近当前后端事实，改动面最小；若未来需要分页，再新开版本化契约。 |
| `DDQ-02` | `Newsletter / Main Channel 来源链接`；`IA-BE-006`、`IA-SPEC-005` | P1 | 原文跳转可用，但 click 行为完全没有回流学习系统。 | 阅读原文是否必须记录为学习信号？若上报失败，是否允许继续跳转？ | A. 先阻塞上报成功再跳转。<br>B. 先发非阻塞埋点（如 beacon / fire-and-forget），无论结果都允许跳转。<br>C. 不记录 click。 | **B（推荐）**：保留阅读主路径流畅性，同时把 click 纳入学习信号；失败只记录日志，不打断跳转。 |
| `DDQ-03` | `Newsletter / Main Channel 点赞 / 点踩按钮`；`IA-BE-005`、`IA-SPEC-005` | P0 | UI 已展示反馈控件，但没有任何提交语义与接口映射。 | 显式反馈应采用什么交互与数据规则进入学习闭环？ | A. 保留 inline like / dislike，点击即提交二元反馈。<br>B. 点击后再弹二级面板收集原因。<br>C. 在能力未接线前隐藏 / 禁用反馈按钮。 | **A（推荐）**：先落稳二元反馈闭环（对象 ID、action、成功/失败提示、去重规则）；若暂时无法接线，则按 C 处理，不能继续冒充可用。 |
| `DDQ-04` | `InterestSettings / 兴趣卡片 Delete 操作`；`IA-BE-002` | P1 | 后端 DELETE 返回 `204`，前端通用封装仍强制 `res.json()`。 | 删除成功的统一契约应该是什么？ | A. 保留 `204 No Content`，前端统一把空响应视为成功。<br>B. 所有 DELETE 都返回 JSON 成功体。 | **A（推荐）**：更符合当前 REST 语义；关键是把“204 也算成功”收敛到统一 fetch 封装，避免每个页面各自兜底。 |
| `DDQ-05` | `Settings / Settings 页面加载`；`IA-SPEC-004` | P1 | 加载失败时页面静默回退默认值，用户看不出当前并未成功读取配置。 | 设置页加载失败 / 未配置时，界面应该如何分流？ | A. 失败时完全阻塞，只允许 Retry。<br>B. 明确区分加载中、加载失败、未配置、新建配置四种状态，并给出下一步入口。<br>C. 继续沿用静默默认值。 | **B（推荐）**：既避免把默认值误当成已保存配置，也允许用户在明确上下文下继续进入“新建配置”流程。 |
| `DDQ-06` | `Settings / Test Connection 按钮`；`IA-BE-007`、`IA-SPEC-003` | P1 | 当前测试的是“后端已保存配置”，不是用户刚修改的表单。 | `Test Connection` 默认应该验证哪一个对象？ | A. 只测已保存配置。<br>B. 只测当前表单草稿。<br>C. 同时提供“测试当前草稿”和“测试已保存配置”两个动作。 | **B（推荐）**：最符合用户心智模型；若保留 A，应降级为诊断动作，不应继续占据主按钮语义。 |
| `DDQ-07` | `Settings / Save Architecture 按钮`；`IA-BE-007`、`IA-SPEC-003` | P1 | 用户未重填 `api_key` 时，保存可能把已有密钥覆盖为空。 | 保存配置时，空 `api_key` 应该表示什么？ | A. 空值表示“保留已有密钥”，若要清除必须单独触发明确动作。<br>B. 空值表示“清空密钥”。<br>C. 每次保存都必须重输密钥。 | **A（推荐）**：风险最低，也最符合配置编辑场景；PRD 必须同步写清表单文案、接口字段语义与显式清除入口。 |
