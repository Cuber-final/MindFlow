> 说明：本文档基于现有 README、技术规格与开发计划反推整理，聚焦产品定位、需求范围、功能结构与关键交互，不包含 UI 风格指导。

## 1. 文档目的

本文件的目标不是描述技术实现，也不是给出视觉风格规范，而是为 UI 设计师提供清晰的产品输入：
- 这个产品是什么
- 为谁服务
- 用户要完成什么任务
- 系统需要提供哪些能力
- 哪些交互是核心亮点
- 哪些内容需要在界面中被重点表达

设计师应基于这份文档理解产品本质，再决定页面结构、信息组织和交互表达。

## 2. 产品概述

### 2.1 产品名称

MindFlow

### 2.2 一句话定位

一个面向个人私有化部署用户的 AI 每日资讯简报系统。

### 2.3 产品本质

MindFlow 不是传统资讯阅读器，也不是简单的 RSS 聚合器。

它的核心价值在于：
- 用户自己选择信息源
- 系统自动抓取内容
- AI 对内容进行观点级加工，而不仅是摘要
- 系统结合用户兴趣做个性化简报编排
- 用户通过反馈持续训练推荐结果

换句话说，MindFlow 提供的不是“文章列表”，而是“由系统协助生成的个人每日观察简报”。

### 2.4 产品形态

产品以浏览器端为主要使用入口，支持本地或自建环境部署，核心使用场景为：
- 每天定时生成一份可阅读的简报
- 用户在简报中快速浏览重点内容
- 用户通过轻交互表达偏好
- 用户按需管理兴趣、信息源和 AI 配置

### 2.5 页面清单总览

为了方便 UI 设计师快速聚焦，本产品当前版本可明确收敛为 6 类页面或页面状态：

1. 简报首页
系统主入口，展示当天或指定日期的每日简报，是最核心的阅读页面。

2. 历史简报查看
本质上属于简报页的日期切换能力，用于查看过去某一天的简报内容。

3. 兴趣标签管理页
用于查看、添加、冻结、删除兴趣标签，并理解推荐权重变化。

4. 信息源管理页
用于管理内容来源，包括添加、编辑、删除、手动抓取和查看来源状态。

5. AI 配置页
用于配置 AI 提供商、接口地址、模型和 Key，并验证连接是否可用。

6. 关键状态页或状态模块
包括加载中、空状态、配置未完成、抓取失败、测试失败等场景，用于保证系统在非理想状态下依然可理解、可行动。

### 2.6 页面职责说明

从产品结构上，这 6 类页面不是彼此孤立的，而是构成一个完整闭环：

- 简报首页：输出结果，回答“今天值得看什么”
- 历史简报查看：扩展阅读，回答“过去发生了什么”
- 兴趣标签管理页：偏好控制，回答“系统应该更关注什么”
- 信息源管理页：输入控制，回答“系统应该从哪里获取内容”
- AI 配置页：能力控制，回答“系统用什么方式理解内容”
- 关键状态页或状态模块：流程兜底，回答“当系统尚未完成配置或运行失败时，用户下一步该做什么”

## 3. 目标用户

### 3.1 核心用户画像

当前产品主要面向以下人群：

1. 高频信息摄入者
需要每天追踪多个主题，如 AI、科技、商业、投资、研究等，但不希望被原始信息流淹没。

2. 注重观点质量的人
不满足于“标题 + 摘要”，更希望系统帮助提炼核心观点，并给出支持、质疑、延伸等更有判断力的内容。

3. 对数据控制有要求的人
希望信息源、阅读记录、偏好数据保留在本地或私有环境中，不依赖公有云平台。

4. 愿意长期训练系统的人
不是一次性消费资讯，而是希望系统随着使用逐步更懂自己。

### 3.2 用户特征

这类用户通常具备以下特征：
- 有稳定的主题关注方向
- 有长期阅读习惯
- 愿意花少量时间配置系统，换取后续更高的信息效率
- 更看重内容组织质量，而非社交互动

## 4. 核心问题

MindFlow 需要解决的核心问题有四个：

### 4.1 信息过载

用户订阅和关注的信息源越来越多，但没有足够时间逐篇阅读。

### 4.2 传统聚合器只做收集，不做理解

很多产品只能把文章堆在一起，缺少观点提炼、主题归纳和判断辅助。

### 4.3 推荐系统通常不透明、不可控

常见信息流平台会推荐内容，但用户很难理解为什么被推荐，也难以主动调整。

### 4.4 长期阅读偏好会变化

用户关注的主题会持续变化，需要一个能学习、但又允许用户主动干预的系统。

## 5. 产品目标

### 5.1 核心目标

1. 每天为用户生成一份结构化、可快速阅读的资讯简报
2. 帮助用户从海量原始文章中直接看到“值得看的观点”
3. 通过兴趣标签和行为反馈实现个性化内容编排
4. 在个性化推荐的同时，保留探索和意外发现
5. 让用户对信息源、AI 配置与兴趣方向保持控制权

### 5.2 体验目标

1. 用户打开系统后，应能在短时间内理解“今天最值得看的内容是什么”
2. 用户应能直观理解一条内容为什么会出现在简报里
3. 用户调整兴趣和信息源后，应能感知系统在下一轮简报中发生变化
4. 用户应感到系统是在“协助整理世界”，而不是“继续推送噪音”

### 5.3 非功能性目标

基于技术文档，产品层面对应的体验要求为：
- 每日简报生成应足够快，能够支持日常使用
- 页面加载要轻量，不应让用户等待过久
- 推荐权重调整要稳定，不能出现剧烈波动

## 6. 产品边界

### 6.1 当前版本要做的事

当前版本聚焦以下闭环：
- 录入并管理信息源
- 抓取文章
- AI 提炼锚点与辩证分析
- 生成每日简报
- 管理兴趣标签
- 记录显式与隐式反馈
- 根据反馈更新兴趣权重
- 查看和切换历史简报
- 配置 AI 服务能力

### 6.2 当前版本不重点解决的事

以下能力暂不属于核心目标：
- 社交分享与社区讨论
- 多用户协作
- 内容评论系统
- 复杂权限体系
- 移动端原生应用
- 面向公开读者的内容发布

### 6.3 规划中但不必在当前设计中强绑定的能力

文档中提到但处于后续扩展阶段的能力包括：
- 知乎内容接入
- RSS 内容接入
- 更多信息源类型

设计上应预留扩展空间，但不需要为未实现能力设计过度复杂的交互。

## 7. 产品核心机制

### 7.1 内容加工机制

系统不是直接展示原文，而是先做内容加工：

1. 抓取原始文章
2. 提炼“锚点”
3. 生成辩证分析
4. 打上主题标签
5. 按兴趣与探索策略编排进简报

对于设计师来说，这意味着前端展示的核心对象不是“文章”，而是“加工后的洞察单元”。

### 7.2 个性化机制

产品通过两类信号学习用户偏好：
- 显式反馈：如 show / hide
- 隐式反馈：如停留、点击、滚动、回访

设计上需要让用户感知到：
- 系统在学习
- 学习依据和兴趣标签有关
- 用户可以主动修正系统方向

### 7.3 探索机制

系统并非只投喂强兴趣内容，而是采用三层内容结构：
- 主航道：最符合当前兴趣的核心内容
- 探索区：可能相关但尚未充分验证的新方向
- 惊喜箱：弱相关但值得打破信息茧房的内容

这是产品差异化能力之一，必须在产品信息结构上被明确体现。

## 8. 核心对象定义

为了让设计师准确理解页面里的信息层级，产品层面有五个核心对象：

### 8.1 简报

简报是按日期生成的一份每日内容集合，包含：
- 标题
- 日期
- 导语
- 分组后的洞察内容
- 总处理文章数
- 锚点总数

### 8.2 洞察

洞察是系统提炼后的最小内容单位，包含：
- 标题
- 核心内容
- 辩证分析
- 来源
- 标签
- 所属内容分区

### 8.3 兴趣标签

兴趣标签代表用户当前的关注主题，具有：
- 标签名称
- 权重
- 状态
- 展示 / 点击 / 阅读等统计数据

### 8.4 信息源

信息源是内容进入系统的入口，具有：
- 名称
- 类型
- 配置参数
- 抓取状态
- 文章数量
- 上次抓取时间

### 8.5 AI 配置

AI 配置是系统内容加工能力的基础设置，主要包括：
- 提供商
- API Key
- Base URL
- 模型

## 9. 用户任务拆解

从用户视角看，产品需要支持的核心任务如下：

### 9.1 每日阅读任务

用户希望：
- 打开系统就看到当天简报
- 快速知道今天有哪些重点主题
- 判断哪些内容值得深入看
- 通过阅读和点击完成一轮轻量消费

### 9.2 回看历史任务

用户希望：
- 切换到某一天的简报
- 回看过去一段时间的内容变化
- 对比不同日期的主题分布

### 9.3 调整兴趣任务

用户希望：
- 添加一个自己关心的新主题
- 冻结或删除不再关心的主题
- 理解某个主题目前权重高低
- 感知系统是否学到了自己的偏好

### 9.4 管理信息源任务

用户希望：
- 添加新的内容来源
- 编辑已有来源参数
- 删除无效来源
- 主动触发抓取，验证来源是否可用

### 9.5 管理系统能力任务

用户希望：
- 配置 AI 提供商和模型
- 测试配置是否可用
- 确认系统当前运行状态和基础能力

## 10. 功能需求

以下功能需求按页面组织，便于 UI 设计师直接映射为页面设计任务。

### 10.1 简报首页

### 功能目标

简报首页是整个产品的主入口，负责承载“今天值得看什么”。

### 必备能力

1. 展示最新一份简报
2. 支持切换历史日期
3. 展示简报标题、导语、日期和简报统计
4. 按主题或领域分组展示内容
5. 展示每条洞察的标题、内容、辩证分析、来源、标签、分区
6. 支持按标签筛选当前简报中的内容
7. 支持点击原文
8. 支持记录阅读行为

### 设计重点

设计师需要把这页理解为“每日阅读入口”，不是列表页。

重点需要突出：
- 今日主题感
- 内容分组逻辑
- 每条洞察的可读性
- 主航道 / 探索区 / 惊喜箱的差异
- 日期切换与历史访问

### 10.2 历史简报切换

### 功能目标

用户能够把系统当作“按天归档的内容观察记录”来使用。

### 必备能力

1. 查看已有简报日期列表
2. 快速切换到某个日期
3. 保持切换过程简单，不打断阅读

### 设计重点

历史切换应是阅读流程的一部分，而不是跳去另一个管理页。

### 10.3 兴趣标签管理

### 功能目标

用户需要一个能理解并影响推荐结果的地方。

### 必备能力

1. 查看所有兴趣标签
2. 添加新标签
3. 删除标签
4. 冻结 / 激活标签
5. 查看标签权重
6. 查看标签统计信息
7. 查看系统推荐的新标签候选
8. 一键采纳候选标签

### 设计重点

这页不是纯 CRUD 页面，而是“推荐逻辑解释页”。

用户应该在这里感知：
- 系统目前重点关注哪些方向
- 某个方向为什么重要或不重要
- 自己如何改变下一份简报

### 10.4 信息源管理

### 功能目标

用户需要清楚掌控“内容从哪里来”。

### 必备能力

1. 查看全部信息源
2. 添加信息源
3. 从链接尝试添加信息源
4. 编辑信息源
5. 删除信息源
6. 手动触发抓取
7. 查看来源类型、文章数量、上次抓取时间等状态

### 设计重点

这页强调的是“来源可控”和“系统可靠性”，不是内容阅读。

UI 需要让用户容易判断：
- 这个来源是否有效
- 系统最近有没有成功抓取
- 添加一个来源的成本是否低

### 10.5 AI 配置

### 功能目标

用户需要配置系统的大脑。

### 必备能力

1. 查看当前 AI 提供商配置
2. 修改提供商、API 地址、模型
3. 输入或更新 API Key
4. 测试连接是否成功
5. 保存配置

### 设计重点

这页最重要的不是复杂功能，而是：
- 降低配置门槛
- 提升可理解性
- 让用户明确知道“是否配置成功”

### 10.6 空状态、错误状态与加载状态

### 功能目标

产品必须在“还没数据”或“某一步失败”时仍然可理解。

### 必备状态

1. 无简报数据
2. 无兴趣标签
3. 无信息源
4. AI 配置未完成
5. 抓取失败
6. 测试连接失败
7. 加载中

### 设计重点

空状态不只是提示“没有内容”，还应告诉用户下一步应该做什么。

## 11. 关键交互亮点

以下不是纯功能点，而是设计时必须强化的产品亮点。

### 11.1 今日简报感

用户进入系统后，应感到自己是在阅读一份“今日整理好的内容”，而不是在翻数据库。

### 11.2 洞察而非摘要

系统展示的是观点级内容单元，而不是文章截断摘要。设计中要帮助用户理解这点。

### 11.3 辩证分析

每条内容不仅提供“这篇文章说了什么”，还提供：
- 支持依据
- 质疑角度
- 延伸思考

这部分是产品差异化重点之一。

### 11.4 个性化可感知

用户应能明确感知：
- 这份简报和兴趣标签有关
- 反馈会影响后续结果
- 推荐不是黑箱

### 11.5 探索与反茧房

产品不应该只重复用户已经知道的内容，而要保留探索空间。

“主航道 / 探索区 / 惊喜箱”的分层，既是内容机制，也是交互亮点。

### 11.6 配置与内容闭环

从“添加信息源”到“生成简报”再到“调整兴趣”，应让用户感受到这是一个闭环系统，而不是分裂的多个工具页。

## 12. 页面层级与信息架构建议

本节不是视觉规范，而是信息架构输入。

### 12.1 一级页面

建议一级信息结构围绕四个任务组织：
- 简报
- 兴趣
- 信息源
- 设置

### 12.2 页面关系

页面关系应让用户理解：
- 简报是结果页
- 兴趣是偏好控制页
- 信息源是输入管理页
- 设置是系统能力页

四者形成因果关系，而不是平行孤立的四个页面。

### 12.3 信息优先级

全站信息优先级建议如下：

1. 今日简报内容
2. 推荐逻辑相关信息（标签、分区、反馈）
3. 来源与系统状态
4. 配置项本身

---

## 13. 给 UI 设计师的设计输入重点

下面这些内容不是风格要求，而是产品信息表达重点：

### 13.1 必须被设计强化的信息

1. 今天的简报是“生成结果”
2. 洞察是“加工后内容”，不是原文
3. 辩证分析是差异化能力
4. 兴趣标签是推荐系统的控制面板
5. 主航道 / 探索区 / 惊喜箱是核心内容结构
6. 信息源管理决定系统“看什么”
7. AI 配置决定系统“怎么理解内容”

### 13.2 设计上应帮助回答的问题

一个好的界面，应该能帮助用户快速回答以下问题：

1. 今天最值得看什么？
2. 这条内容为什么会出现？
3. 这是我的主兴趣，还是系统在探索新方向？
4. 我怎样让下一份简报更符合我的需求？
5. 内容来源是否可信、系统是否工作正常？

## 14. 版本优先级建议

### P0：必须在首版设计中完整体现

1. 简报阅读页
2. 洞察单元与辩证分析
3. 日期切换
4. 兴趣标签管理
5. 信息源管理
6. AI 配置
7. 空状态 / 加载状态 / 错误状态

### P1：应在结构上预留，但可后续增强

1. 更多信息源类型
2. 更多行为反馈可视化
3. 历史学习轨迹展示
4. 更丰富的探索机制表达

### P2：暂不纳入当前重点

1. 社交传播
2. 多用户体系
3. 复杂权限与团队协作
4. 内容生产或编辑工作流

---

## 15. 成功标准

如果 UI 设计成功，用户应能在首次使用或短时间试用后形成以下认知：

1. 这是一个帮我整理每天信息输入的工具
2. 它会根据我的兴趣生成简报，而不是简单推送文章
3. 我可以主动控制来源、兴趣和系统配置
4. 我能理解它是如何逐步变得更懂我的

如果用户只感受到“这是一个文章后台”或“这是一个信息源管理工具”，说明产品表达仍然不完整。

---

## 16. 附录：一句话给设计师

请把 MindFlow 设计成一个“个人每日世界观察系统”。

它的核心不是资讯堆叠，也不是后台操作，而是让用户每天用最少的时间，获得一份更懂自己、但又不过度封闭的信息简报。
